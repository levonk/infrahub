# Message Queue Architecture

## Overview

The message queue system provides asynchronous processing for AI analytics events, enabling the hot path (request forwarding) to remain fast while heavy analytics processing happens in the background.

## Architecture Components

### 1. Redis Client (`redis_client.py`)
- **Purpose**: Manages Redis connections with pooling and automatic reconnection
- **Key Features**:
  - Connection pooling for performance
  - Automatic reconnection with exponential backoff
  - Health monitoring via ping
  - Thread-safe operations
  - Context manager support

### 2. Queue Producer (`producer.py`)
- **Purpose**: Enqueues analytics events from collectors
- **Key Features**:
  - Non-blocking message enqueue
  - Message serialization (JSON)
  - Batch enqueue support
  - Error handling with graceful degradation
  - Integration with collector framework

### 3. Queue Consumer (`consumer.py`)
- **Purpose**: Dequeues and processes messages from Redis Streams
- **Key Features**:
  - Redis Streams consumer group support
  - Batch message processing
  - Automatic acknowledgment
  - Dead letter queue routing
  - Blocking reads with timeout

### 4. Processor Worker (`worker.py`)
- **Purpose**: Individual worker that processes analytics events
- **Key Features**:
  - Event deserialization
  - Database persistence
  - Error handling and retry logic
  - Processing time tracking
  - Graceful shutdown support

### 5. Worker Pool (`pool.py`)
- **Purpose**: Manages multiple worker processes
- **Key Features**:
  - Dynamic worker scaling
  - Load balancing
  - Health monitoring
  - Graceful shutdown coordination
  - Worker utilization tracking

### 6. Configuration (`config.py`)
- **Purpose**: Centralized configuration management
- **Key Features**:
  - Environment variable support
  - Default values for all parameters
  - Type-safe configuration
  - Separate configs for Redis, Queue, and Processor

### 7. Metrics (`metrics.py`)
- **Purpose**: Collects and reports queue performance metrics
- **Key Features**:
  - Message throughput tracking
  - Error rate monitoring
  - Queue depth monitoring
  - Worker utilization tracking
  - Health status calculation

### 8. Error Handling (`errors.py`)
- **Purpose**: Custom exceptions for queue operations
- **Key Features**:
  - Specific exception types for different failure modes
  - Clear error categorization
  - Support for retry logic

## Data Flow

```
Collector → Producer → Redis Stream → Consumer → Worker Pool → Database
                ↓              ↓
            Metrics ← Dead Letter Queue
```

### Hot Path (Request Forwarding)
1. Collector intercepts HTTP request
2. Extracts metadata (RequestMetadata, ResponseMetadata)
3. Creates AnalyticsEvent
4. Enqueues event via Producer (non-blocking)
5. Forwards request to target (minimal latency)

### Cold Path (Analytics Processing)
1. Consumer reads messages from Redis Stream
2. Worker pool processes events
3. Persists to SQLite database
4. Updates metrics
5. Handles errors with retry logic

## Redis Streams Design

### Stream Structure
- **Stream Name**: `analytics_events`
- **Consumer Group**: `analytics_processors`
- **Message Format**: JSON-serialized AnalyticsEvent
- **Message ID**: Auto-generated by Redis (timestamp-sequence)

### Consumer Group Benefits
- Multiple consumers can read from same stream
- Each message processed by exactly one consumer
- Automatic message acknowledgment
- Pending message tracking for recovery

### Dead Letter Queue
- **Stream Name**: `analytics_events_dlq`
- **Purpose**: Stores permanently failed messages
- **Triggers**: Max retry attempts exceeded
- **Monitoring**: Separate metrics for DLQ

## Performance Considerations

### Latency Targets
- Producer enqueue: <1ms (non-blocking)
- Consumer dequeue: <5ms (batch processing)
- Worker processing: <100ms per event
- End-to-end: <1s for typical analytics

### Throughput Targets
- Producer: 10,000+ messages/second
- Consumer: 5,000+ messages/second
- Worker pool: Scales with worker count

### Resource Limits
- Max message size: 1MB
- Max queue depth: Configurable (warning at 1000, critical at 5000)
- Worker count: Configurable (default 4)
- Redis connections: Configurable (default 50)

## Reliability Features

### Connection Reliability
- Connection pooling
- Automatic reconnection with backoff
- Health checks via ping
- Connection timeout handling

### Message Reliability
- Redis Streams persistence (AOF/RDB)
- Consumer group acknowledgments
- Retry logic with exponential backoff
- Dead letter queue for failed messages

### Processing Reliability
- Worker pool redundancy
- Graceful shutdown (in-flight messages)
- Error isolation (one worker failure doesn't affect others)
- Processing time tracking

## Security Considerations

- Use references instead of raw content
- Message size limits enforced
- Configurable retention policies

### Access Control
